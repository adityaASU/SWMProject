# System Architecture

## Overview

The Text2SQL LRG platform has four distinct layers, each fully decoupled and independently testable.

```
┌────────────────────────────────────────────────────────────┐
│                     Interface Layer                        │
│   CLI scripts   │   FastAPI REST   │   Streamlit UI        │
└────────────────────────────────────────────────────────────┘
                            │
┌────────────────────────────────────────────────────────────┐
│                   Baseline / Model Layer                   │
│   BaseText2SQL (ABC)                                       │
│   ├── PromptBaseline (zero-shot / few-shot)                │
│   └── LRGText2SQL (full pipeline)                          │
└────────────────────────────────────────────────────────────┘
                            │
┌────────────────────────────────────────────────────────────┐
│                   LRG Core Layer                           │
│   LRGBuilder  ──►  LRGGraph  ──►  SQLSynthesizer          │
│   (LLM calls)      (DAG)         (deterministic)          │
└────────────────────────────────────────────────────────────┘
                            │
┌────────────────────────────────────────────────────────────┐
│                 Foundation Layer                           │
│   LLM Backends │ Schema Parser │ Evaluation │ Benchmark   │
└────────────────────────────────────────────────────────────┘
```

## Data Flow

1. **Input**: natural language question + database ID
2. **Schema parsing**: `SchemaParser.auto_parse(db_dir, db_id)` reads SQLite or Spider JSON
3. **Schema graph**: `SchemaGraph(schema)` builds a NetworkX DiGraph of tables and FK edges
4. **LRG construction** (LRG model only):
   - `LRGBuilder.build()` sends schema + question to LLM as structured JSON extraction
   - LLM returns entities, filters, aggregations, join hints, etc.
   - Python assembles the `LRGGraph`, validating all join paths against the schema graph
5. **SQL synthesis**: `SQLSynthesizer.synthesize(lrg)` traverses the graph deterministically — no LLM
6. **Evaluation**: `BenchmarkRunner` compares predictions to gold SQL using EM, EX, failure mode analysis, and explainability metrics
7. **Output**: JSON + Markdown reports, optional LRG PNG visualisation

## Key Design Decisions

### Why Two-Stage LRG?

The LLM is asked to produce **structured JSON** (not SQL) — a much simpler output space. Python then assembles the LRGGraph and validates all join paths against the schema graph. SQL is only generated by the deterministic synthesizer. This separates "what to compute" (LLM task) from "how to express it as SQL" (Python task).

### Why NetworkX for the Schema Graph?

NetworkX provides shortest-path algorithms, cycle detection, and bidirectional traversal out of the box. All join path queries reduce to `nx.shortest_path()` calls over the undirected FK graph.

### Why Pydantic for LRG Nodes?

Pydantic v2 models serialise to JSON natively, which enables:
- Storing LRG graphs in benchmark report JSON
- Sending LRG data through the API
- Reconstructing graphs with `LRGGraph.from_dict()`
- Using models directly as LLM output schemas
